# Day 007: Package an ML Project as an Installable Python Package

**Category**: Python Packaging / Distribution | **Difficulty**: 🟡 Intermediate

---

## Problem

The xFusionCorp Industries deployment team needs the fraud-detection model code packaged as an installable Python distribution. A draft `pyproject.toml` exists at `/root/code/fraud-detection/`, but it does not build a wheel that meets the team's standard. Correct the file and produce a compliant package.

Given context:
1. The project at `/root/code/fraud-detection/` already contains the source code under `src/fraud_detection/`. The Python files are complete — no need to modify any of them.
2. The corrected `pyproject.toml` must satisfy every one of the following:
   - it declares a `[build-system]` section with `requires = ["setuptools>=61.0", "wheel"]` and `build-backend = "setuptools.build_meta"`
   - `name` is `fraud_detection`
   - `version` is `0.1.0`
   - `requires-python` is `>=3.10`
   - `dependencies` is `["scikit-learn", "pandas", "numpy"]`
3. Build the package from the project directory:
```bash
cd /root/code/fraud-detection
python3 -m build
```
4. The build must produce a wheel named `fraud_detection-0.1.0-*.whl` under `dist/`

---

## Solution

### Step 1: Inspect the broken pyproject.toml

```bash
cat /root/code/fraud-detection/pyproject.toml
```

### Step 2: Fix pyproject.toml

```bash
nano /root/code/fraud-detection/pyproject.toml 
```

Edit it so it look exactly like this:

```python
[build-system]
requires = ["setuptools>=61.0", "wheel"]
build-backend = "setuptools.build_meta"

[project]
name = "fraud_detection"
version = "0.1.0"
requires-python = ">=3.10"
dependencies = ["scikit-learn", "pandas", "numpy"]

[tool.setuptools.packages.find]
where = ["src"]
```

### Step 3: Build the package

```bash
cd /root/code/fraud-detection
python3 -m build
```

### Step 4: Verify the wheel was created

```bash
ls dist/
```

You should see `fraud_detection-0.1.0-*.whl`.

## Key Takeaways

### 📦 What is a Python Package / Wheel?
A Python **wheel** (`.whl`) is a prebuilt distribution format — a zip file containing your code and metadata, ready to be installed with `pip install`. Packaging your ML code as a wheel means anyone on the team (or any server) can install it with a single command: `pip install fraud_detection-0.1.0-*.whl`. This is how production ML code gets deployed cleanly without copying folders around.

### 🗂️ `pyproject.toml` — the modern packaging standard
`pyproject.toml` is the single file that defines everything about your Python project: its name, version, dependencies, and how to build it. It replaced the old `setup.py` and `setup.cfg` approach. Every modern Python tool reads from this file, making it the source of truth for your project's identity and build process.

### 🔧 `[build-system]` — the most important section
The `[build-system]` section tells Python **how** to build your package. Without it, `python3 -m build` doesn't know which tool to use. It has two keys:
- `requires` — the build tools that need to be installed before building (setuptools, wheel)
- `build-backend` — the Python object that performs the actual build (`setuptools.build_meta`)

This section must be separate from `[project]` — mixing them is a common mistake that silently breaks the build.

### 📁 `src/` layout — why put code inside `src/`?
The `src/` layout is a best practice where your package code lives inside a `src/` folder instead of at the project root. This prevents accidental imports of your local code during testing (you always test the installed version, not the raw files) and makes it explicit what gets packaged. The `[tool.setuptools.packages.find]` section tells setuptools to look inside `src/` for packages to include.
