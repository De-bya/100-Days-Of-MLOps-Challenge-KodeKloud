# Day 012: Configure a DVC Remote Storage

**Category**: Data Version Control | **Difficulty**: 🟢 Beginner

---

## Problem

The xFusionCorp Industries ML team uses SeaweedFS as the shared S3-compatible object store for DVC-tracked data. A `.dvc/config` already declares a remote called `s3` for the fraud-detection project, but `dvc push` currently fails. Correct the configuration and push the tracked data into the SeaweedFS bucket.

Given context:
1. A project exists at `/root/code/fraud-detection/` with DVC initialised and `data/raw/transactions.csv` already tracked.
2. SeaweedFS is running on the controlplane:
   - S3 endpoint: `http://localhost:8333`
   - Credentials: `weedadmin` / `weedadmin123`
   - Bucket name: `dvc-storage` (already created)
3. The remote called `s3` must:
   - point at the `dvc-storage` bucket using `s3://`
   - use the correct SeaweedFS S3 endpoint URL
   - be marked as the default remote
4. After the push, the `dvc-storage` bucket must contain at least one object under the `files/md5/...` prefix.

---

## Solution

### Step 1: Inspect the broken config

```bash
cat /root/code/fraud-detection/.dvc/config
```

### Step 2: Fix the bucket URL

```bash
cd /root/code/fraud-detection
dvc remote modify s3 url s3://dvc-storage
```

### Step 3: Fix the endpoint URL

```bash
dvc remote modify s3 endpointurl http://localhost:8333
```

### Step 4: Set s3 as the default remote

```bash
dvc remote default s3
```

### Step 5: Push the data

```bash
dvc push
```

### Step 6: Verify the config

```bash
cat .dvc/config
```

## Key Takeaways

### ☁️ What is a DVC remote?
A DVC remote is the storage backend where the actual data files live — the equivalent of a `git remote` but for data. While Git pushes code to GitHub, DVC pushes data to a remote storage like S3, GCS, Azure Blob, or any S3-compatible store. Without a remote configured, DVC only caches data locally and teammates have no way to share it.

### 🪣 What is SeaweedFS?
SeaweedFS is an open-source, S3-compatible distributed object store. "S3-compatible" means it speaks the same API as Amazon S3, so tools like DVC, MLflow, and boto3 can talk to it without any code changes — just point them at a different endpoint URL. Teams use it to run a private S3-like storage on their own infrastructure without depending on AWS.

### 🔧 DVC remote configuration commands

```bash
# Add a new remote
dvc remote add <name> s3://<bucket>

# Modify an existing remote setting
dvc remote modify <name> url s3://<bucket>
dvc remote modify <name> endpointurl http://localhost:8333
dvc remote modify <name> access_key_id <key>
dvc remote modify <name> secret_access_key <secret>

# Set as default remote (used by dvc push/pull without -r flag)
dvc remote default <name>
```

### 📁 What `dvc push` actually does
When you run `dvc push`, DVC:
1. Reads every `.dvc` pointer file in the project
2. Finds the corresponding cached files in `.dvc/cache/` using their checksums
3. Uploads them to the remote storage under `files/md5/<checksum>` path

This is why the bucket shows objects under `files/md5/...` — that's DVC's content-addressable storage layout. The same file with the same content always maps to the same path, preventing duplicates.

### 🔄 `dvc push` and `dvc pull` — the data sync workflow

```bash
dvc push   # upload local cache to remote (share your data)
dvc pull   # download from remote to local cache (get teammate's data)
dvc fetch  # download to cache only, don't extract to workspace
```

Think of it exactly like `git push` and `git pull` but for data files.

### 🌐 The default remote
Setting a default remote with `dvc remote default <name>` means `dvc push` and `dvc pull` work without specifying `-r <name>` every time. In a project with multiple remotes (e.g. a fast local NAS and a slow cold S3 archive), the default is the one used in everyday development.
