# Day 010: Install and Initialize DVC in an ML Project

**Category**: Data Version Control | **Difficulty**: 🟢 Beginner

---

## Problem

The xFusionCorp Industries ML team is adopting DVC so that datasets and model files are versioned separately from code. Initialise DVC inside the existing Git repository at `/root/code/fraud-detection/` and record the initialisation in Git.

1. A Git repository already exists at `/root/code/fraud-detection/` with an initial commit.
2. Initialise DVC inside that repository so that the standard `.dvc/` control directory and `.dvcignore` file are created alongside the existing Git working tree.
3. Stage every file that DVC produces during initialisation, and record them in a new Git commit with the message `Initialize DVC`.

---

## Solution

### Step 1: Navigate to the repository

```bash
cd /root/code/fraud-detection
```

### Step 2: Initialise DVC

```bash
dvc init
```

This creates the `.dvc/` directory and `.dvcignore` file inside the Git repo.

### Step 3: Stage the DVC files

```bash
git add .dvc .dvcignore
```

### Step 4: Commit with the exact message

```bash
git commit -m "Initialize DVC"
```

## Key Takeaways

### 🗄️ What is DVC?
DVC (Data Version Control) is an open-source tool built on top of Git that adds versioning for large files — datasets, model weights, pipeline outputs — that Git itself handles poorly. Git is designed for code (text files); it breaks down with multi-gigabyte CSV files or model binaries. DVC solves this by storing a small pointer file in Git while keeping the actual data in a separate storage backend (local disk, S3, GCS, Azure, etc.).

### 🤝 DVC + Git — how they work together
DVC doesn't replace Git — it extends it. Every DVC operation produces or modifies files that are then committed to Git normally. The mental model is:

```
Git  → versions your code, configs, and DVC pointer files
DVC  → versions your data, models, and pipeline outputs
```

When a teammate clones your repo, they get the code instantly via `git clone`, then pull the data separately via `dvc pull`. This keeps the Git repo small and fast while still giving full reproducibility.

### 📁 What `dvc init` creates
Running `dvc init` produces two things inside your Git repo:

| File/Dir | Purpose |
|----------|---------|
| `.dvc/` | DVC's internal control directory — stores config, cache settings, and tmp files |
| `.dvcignore` | Like `.gitignore` but for DVC — tells DVC which files to ignore when scanning |

Both are meant to be committed to Git, which is why we immediately `git add` and `git commit` them.

### 🔄 Why version data separately from code?
In ML projects, data and models are first-class artifacts just like code. Without data versioning:
- You can't reproduce a training run from 3 months ago because the dataset changed
- You can't roll back to a model checkpoint that was overwritten
- Teammates can't share large files through Git without bloating the repository

DVC gives data and models the same reproducibility guarantees that Git gives code.
