# Day 024: Enable MLflow Autologging

**Category**: ML Experimentation | **Difficulty**: 🟢 Beginner

---

## Problem

The xFusionCorp Industries ML team wants to replace the manual `log_param` / `log_metric` boilerplate in their training scripts with MLflow's autologging feature, so every training run captures its constructor parameters, training metrics, and model artefact automatically. A training scaffold has been pre-staged at `/root/code/autolog_experiment.py` — it configures MLflow, fits a small synthetic sklearn model, and prints a confirmation message. Two `# TODO` blocks remain empty.

1. The MLflow tracking server is already running on port `5000`. Only the `Default` experiment is present on first load.
2. Complete the two TODO blocks in `/root/code/autolog_experiment.py` so that:
   - An experiment named `autolog-demo` exists on the MLflow server.
   - At least one run exists in the `autolog-demo` experiment.
   - The run's **Parameters** panel lists every sklearn constructor parameter that `LogisticRegression` implicitly carries (`C`, `max_iter`, `solver`, `tol`, `penalty`, etc.).
   - The **Artifacts** panel on the run contains a model directory with an `MLmodel` descriptor and a pickled estimator.
3. Execute the script with `python3 /root/code/autolog_experiment.py`.

---

## Solution

### Step 1: Inspect the scaffold

```bash
cat /root/code/autolog_experiment.py
```

The two TODO blocks are the only edits needed — both must be in place **before** `model.fit()` runs.

### Step 2: Fill in the two TODOs in VS Code

Open `/root/code/autolog_experiment.py` in the VS Code editor and replace the two TODO comments:

```python
# TODO 1: enable autologging for the sklearn flavour
mlflow.sklearn.autolog()

# TODO 2: set the active experiment
mlflow.set_experiment("autolog-demo")
```

### Step 3: Run the script

```bash
python3 /root/code/autolog_experiment.py
```

Expected output:

```
Autolog run complete — check the MLflow UI
```

The `autolog-demo` experiment now appears in the MLflow UI with one run containing all `LogisticRegression` constructor parameters, metrics, and the serialised model artefact — all captured with zero manual logging calls.

## Key Takeaways

### ⚡ `mlflow.sklearn.autolog()` — zero-boilerplate logging
Calling `mlflow.sklearn.autolog()` registers hooks into sklearn's `fit()` method. When `model.fit()` is called, MLflow automatically captures every constructor parameter passed to the estimator, standard evaluation metrics, and the trained model as an artefact — no `log_param`, `log_metric`, or `log_model` calls needed.

### 📌 Order matters — autolog before fit
Both `mlflow.sklearn.autolog()` and `mlflow.set_experiment()` must be called **before** `model.fit()`. Autolog works by instrumenting the fit call at the moment it happens; if autolog is enabled after fit, nothing is captured. Similarly, `set_experiment` must precede fit so the run lands in the correct experiment.

### 🧪 `mlflow.set_experiment()` — targeting a named experiment
`mlflow.set_experiment("autolog-demo")` sets the active experiment for all subsequent runs in the script. If the experiment does not exist yet, MLflow creates it automatically — no separate `create_experiment` call needed.

### 📦 What autolog captures for sklearn
For a `LogisticRegression`, autolog records every constructor parameter (`C`, `max_iter`, `solver`, `tol`, `penalty`, `random_state`, etc.) — not just the ones explicitly passed. It also logs training metrics and saves the fitted model as an artefact with a full `MLmodel` descriptor, making it immediately loadable via `mlflow.sklearn.load_model()`.

### 🔄 Autolog vs manual logging
Autolog is ideal for quick experimentation and standardised pipelines where all runs use the same estimator family. Manual logging (`log_param`, `log_metric`) remains necessary for custom metrics, non-sklearn frameworks, or when you need fine-grained control over what gets recorded. Both approaches can be combined in the same script.
