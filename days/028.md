# Day 028: Fix a Broken MLflow Project and Re-Run It

**Category**: ML Experimentation | **Difficulty**: 🟡 Intermediate

---

## Problem

The xFusionCorp Industries ML platform team packages their training runs as MLflow Projects so any engineer can reproduce them with a single `mlflow run` invocation. A project has been pre-staged at `/root/code/trainer/`, but the first run from the lab startup already failed — the `MLproject` file carries a subtle command-line bug. Fix the bug and then run the project end to end twice so successful runs are recorded.

1. The MLflow tracking server is already running on port `5000`. The `trainer` experiment already contains a `FAILED` run from the automated run that the lab startup triggered against the broken project.
2. `/root/code/trainer/` contains:
   - `MLproject` — the project descriptor (this file has the bug).
   - `train.py` — the trainer entry point. This file is correct and must not be modified.
3. The fix is confined to `MLproject`. Once repaired, two successful `mlflow run` invocations must be recorded in the `trainer` experiment:
   - One explicit call: `mlflow run . -e train -P n_estimators=200 -P max_depth=10 --env-manager=local` (run from `/root/code/trainer`).
   - One default call: `mlflow run . -e train --env-manager=local`.
4. The end state must include:
   - The `MLproject` command line invokes `train.py` with flag names that match `train.py`'s argparse declarations.
   - The `trainer` experiment contains at least two `FINISHED` runs whose `params.n_estimators` values differ (one at `200`, one at the default `100`).
   - The original `FAILED` run from the startup is still present — it must not be deleted, because the lab's diagnosis trail depends on it.

---

## Solution

### Step 1: Inspect the project files

`train.py` declares its argparse flags as `--n_estimators`, `--max_depth`, `--test_size`, and `--random_seed`, with `allow_abbrev=False` set explicitly to prevent argparse from silently resolving prefix matches.

`MLproject`'s command block passes `--n_est {n_estimators}` instead of `--n_estimators {n_estimators}` — a typo'd flag name. Because abbreviation matching is disabled in `train.py`, this mismatch causes an immediate failure rather than being silently resolved.

### Step 2: Fix the `MLproject` command 

Open `/root/code/trainer/MLproject` and correct the flag name:

```yaml
name: trainer
entry_points:
  train:
    parameters:
      n_estimators:
        type: int
        default: 100
      max_depth:
        type: int
        default: 5
      test_size:
        type: float
        default: 0.2
      random_seed:
        type: int
        default: 42
    command: >
      python train.py
      --n_estimators {n_estimators}
      --max_depth {max_depth}
      --test_size {test_size}
      --random_seed {random_seed}
```

### Step 3: Run the project with explicit parameters

```bash
cd /root/code/trainer
mlflow run . -e train -P n_estimators=200 -P max_depth=10 --env-manager=local
```

### Step 4: Run the project with default parameters

```bash
mlflow run . -e train --env-manager=local
```

The `trainer` experiment now contains the original `FAILED` run plus two new `FINISHED` runs with differing `n_estimators` values.

## Key Takeaways

### 📄 `MLproject` — the project descriptor
The `MLproject` file defines entry points, their parameters (with types and defaults), and the exact shell command MLflow runs to execute them. It is the contract between "how a user invokes the project" and "what command actually runs" — any mismatch between the declared parameters and the command template breaks every invocation.

### 🔒 `allow_abbrev=False` — why the bug didn't get silently fixed
By default, Python's `argparse` resolves unambiguous prefixes (`--n_est` would match `--n_estimators` automatically). Setting `allow_abbrev=False` disables this convenience, forcing exact flag matches. This is good practice for production training scripts — silent prefix resolution can mask genuine typos for years before someone introduces a second flag with the same prefix and breaks everything.

### 🏃 `mlflow run` — reproducible execution
`mlflow run <project_dir> -e <entry_point> -P key=value` runs a declared entry point with parameter overrides, automatically creating a tracking run scoped to whichever experiment is currently active. `--env-manager=local` skips conda/virtualenv environment creation and runs directly in the current Python environment — appropriate when the project's dependencies are already satisfied.

### 🧪 Don't delete failed runs during debugging
The original `FAILED` run was deliberately preserved as part of the task. In real diagnostic workflows, failed runs carry the exact error trace and parameters that triggered the failure — deleting them destroys the evidence needed to confirm a fix actually addresses the root cause, and to demonstrate to teammates what was broken and why.

### 🔍 Isolating the fix to one file
The task constrained the fix to `MLproject` only, with `train.py` declared off-limits. This mirrors a common real-world debugging discipline: when a script is known-correct and a wrapper/config file is suspect, changing only the wrapper keeps the fix minimal and verifiable, rather than risking new bugs by touching working code.
