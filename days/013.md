# Day 013: Pull DVC-Tracked Data from Remote

**Category**: Data Version Control | **Difficulty**: 🟢 Beginner

---

## Problem

A new xFusionCorp Industries team member has cloned the fraud-detection repository onto a fresh machine. The DVC remote is already configured to point at the team's SeaweedFS bucket, but `dvc pull` is failing. Diagnose the cause, correct the configuration, and pull the dataset.

Given context:
1. A cloned project exists at `/root/code/fraud-detection/` with DVC initialised and the `data/raw/transactions.csv.dvc` pointer file present, but the dataset itself is missing from disk and from the local DVC cache.
2. SeaweedFS is running with the dataset already pushed to the `dvc-storage` bucket. S3 endpoint: `http://localhost:8333`, credentials: `weedadmin` / `weedadmin123`.
3. Review `.dvc/config` and correct everything that prevents `dvc pull` from authenticating against SeaweedFS.
4. After the fix, `data/raw/transactions.csv` must be present on disk and its content must match the hash recorded in the `.dvc` pointer.

---

## Solution

### Step 1: Inspect the config

```bash
cat /root/code/fraud-detection/.dvc/config
```

### Step 2: Add the missing credentials

```bash
cd /root/code/fraud-detection
dvc remote modify s3 access_key_id weedadmin
dvc remote modify s3 secret_access_key weedadmin123
```

### Step 3: Pull the data

```bash
dvc pull
```

### Step 4: Verify the file is on disk

```bash
ls data/raw/
```

## Key Takeaways

### 🔑 Credentials are never stored in Git
The `.dvc/config` file is committed to Git — which means it is public (or at least visible to everyone with repo access). Storing passwords or access keys in it would be a serious security risk. This is why credentials are often intentionally left out of the committed config and added locally on each machine by the person setting it up. In production, credentials are injected via environment variables or secrets managers instead.

### 🌍 The "fresh clone" problem in MLOps
When a new teammate clones an ML repo, they get the code and the `.dvc` pointer files — but not the actual data. This is by design. The workflow is always:
```bash
git clone <repo>        # get the code and pointers
dvc pull                # get the actual data from remote storage
```
This is the MLOps equivalent of cloning a repo and running `pip install -r requirements.txt` — two separate steps to get code and dependencies.

### 🔐 DVC credential options
There are three ways to provide credentials to a DVC remote:

| Method | How |
|--------|-----|
| Config file | `dvc remote modify s3 access_key_id <key>` |
| Environment variables | `AWS_ACCESS_KEY_ID=<key> dvc pull` |
| IAM roles (cloud) | No credentials needed — the machine's identity is used |

For local/dev environments, the config file approach is fine. For CI/CD pipelines, environment variables or IAM roles are the standard.

### ✅ How `dvc pull` works
When you run `dvc pull`, DVC:
1. Reads every `.dvc` pointer file in the project
2. Checks if the file with that checksum already exists in the local cache (`.dvc/cache/`)
3. If not, downloads it from the remote storage
4. Links or copies the file from cache into the workspace

This means if two projects use the same dataset, DVC only downloads it once — the second project just links to the cached copy.
