# Day 016: Track ML Metrics with DVC

**Category**: Data Version Control / ML Metrics | **Difficulty**: 🟢 Beginner

---

## Problem

After training a model, the xFusionCorp Industries ML team wants DVC to surface metrics through `dvc metrics show` and the DVC extension's METRICS view. The fraud-detection pipeline already trains a model and writes a `metrics.json`, but DVC does not recognise the file as a metric. Wire it in correctly.

Given context:
1. A project exists at `/root/code/fraud-detection/` with a three-stage DVC pipeline. The `train` stage runs `src/models/train.py`, which writes the model to `models/model.pkl` and metrics to `metrics.json`.
2. The `train` stage in `dvc.yaml` must declare `metrics.json` as a DVC metric output, not as a regular file output. The metric must be declared with `cache: false` so the JSON lives in Git for diff history rather than in the DVC cache.
3. Re-run the pipeline with `dvc repro` so the metric registration takes effect.
4. After your changes, `dvc metrics show` must report the `accuracy` and `f1_score` values from `metrics.json`.

---

## Solution

### Step 1: Inspect the current dvc.yaml

```bash
cat /root/code/fraud-detection/dvc.yaml
```

### Step 2: Fix the train stage in dvc.yaml

```bash
nano /root/code/fraud-detection/dvc.yaml
```

Change the `train` stage to:

```yaml
  train:
    cmd: python src/models/train.py
    deps:
      - data/processed/train.csv
      - src/models/train.py
    outs:
      - models/model.pkl
    metrics:
      - metrics.json:
          cache: false
```

Save with `Ctrl+O` → `Enter` → `Ctrl+X`

### Step 3: Re-run the pipeline

```bash
cd /root/code/fraud-detection
dvc repro
```

### Step 4: View the metrics

```bash
dvc metrics show
```

You should see `accuracy` and `f1_score` reported.

## Key Takeaways

### 📊 `outs` vs `metrics` in dvc.yaml
DVC distinguishes between two types of outputs:

| Type | Purpose | Stored in |
|------|---------|-----------|
| `outs` | Large files (models, datasets) | DVC cache + remote |
| `metrics` | Small JSON/CSV evaluation results | Git (not DVC cache) |

`metrics.json` belongs under `metrics` because it is small, human-readable, and you want to track its history in Git — not ship it to S3.

### 🔓 `cache: false` — why metrics live in Git
By default, DVC caches every output and adds it to `.gitignore`. Setting `cache: false` tells DVC: "don't cache this file, let Git track it directly." This is essential for metrics because:
- You want to `git diff` metrics between commits
- You want the values visible on GitHub without running `dvc pull`
- Tools like `dvc metrics diff` compare values across Git commits

### 📈 `dvc metrics show`
Once `metrics.json` is wired correctly, `dvc metrics show` reads the file and displays all key-value pairs in a clean table:

```
Path          accuracy    f1_score
metrics.json  0.95        0.93
```

This works across branches and commits — `dvc metrics diff main` compares the current metrics against the main branch.

### 🔄 Metrics as first-class citizens in MLOps
Tracking metrics in DVC closes the loop between code, data, parameters, and results. Every Git commit now captures the full experiment snapshot:
- Code → Git
- Data → DVC pointer in Git
- Parameters → `params.yaml` in Git
- Metrics → `metrics.json` in Git
- Model → DVC cache

This is reproducible ML: given any commit, you can recreate the exact experiment and verify the results.
