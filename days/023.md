# Day 023: Search and Tag MLflow Runs

**Category**: ML Experimentation | **Difficulty**: 🟢 Beginner

---

## Problem

A xFusionCorp Industries data scientist has accumulated ten runs in the `fraud-detection` MLflow experiment. Triage those runs via the MLflow UI: mark the single best-performing candidate as the shortlisted model, and flag every clearly under-performing run for removal.

1. The MLflow tracking server is already running on port `5000`, and the `fraud-detection` experiment has been pre-populated with ten runs.
2. Complete the triage below — the end state is what is tested, not the path taken through the UI:
   - **Shortlist the best candidate.** Among all runs where `metrics.f1_score > 0.85`, the single run with the highest `f1_score` must carry a run-level tag: key `review-status`, value `shortlisted`.
   - **Reject the under-performers.** Every run where `metrics.f1_score < 0.75` must carry a run-level tag: key `review-status`, value `rejected`.
3. The other runs (those in the 0.75 ≤ f1 ≤ 0.85 band, and the second-best shortlisting candidate) must carry no `review-status` tag at all.

---

## Solution

### Step 1: Inspect all runs and their f1_score metrics

```bash
python3 << 'EOF'
import mlflow
mlflow.set_tracking_uri("http://localhost:5000")
client = mlflow.tracking.MlflowClient()

exp = client.get_experiment_by_name("fraud-detection")
runs = client.search_runs(exp.experiment_id, order_by=["metrics.f1_score DESC"])
for r in runs:
    print(r.info.run_id, '|', round(r.data.metrics.get("f1_score", 0), 4), '|', r.data.tags)
EOF
```

### Step 2: Apply the tags via the MLflow UI

**Tag the best run as `shortlisted`:**

1. Open the MLflow UI and click the **fraud-detection** experiment.
2. Click the run **rogue-goose-896** (f1=0.95).
3. In the **Tags** section on the run page, click **Add tag**.
4. Enter key: `review-status`, value: `shortlisted` → Save.

**Tag the two under-performers as `rejected`:**

5. Go back to the experiment, click **nebulous-dog-196** (f1=0.72).
6. Add tag → key: `review-status`, value: `rejected` → Save.
7. Go back, click **amazing-shrew-533** (f1=0.70).
8. Add tag → key: `review-status`, value: `rejected` → Save.

### Alternative: Apply all tags programmatically

```bash
python3 << 'EOF'
import mlflow
mlflow.set_tracking_uri("http://localhost:5000")
client = mlflow.tracking.MlflowClient()

# Shortlist the best run (f1=0.95)
client.set_tag("3c2f2a515e3c44ef8892c9e881690794", "review-status", "shortlisted")

# Reject under-performers (f1 < 0.75)
client.set_tag("c6b7e3fc5f16414aa6d8c6a8afee22ad", "review-status", "rejected")
client.set_tag("51baea0c276844449c4df4433b4071bd", "review-status", "rejected")

print("Done.")
EOF
```

### Step 3: Verify the final tag state

```bash
python3 << 'EOF'
import mlflow
mlflow.set_tracking_uri("http://localhost:5000")
client = mlflow.tracking.MlflowClient()

exp = client.get_experiment_by_name("fraud-detection")
runs = client.search_runs(exp.experiment_id, order_by=["metrics.f1_score DESC"])
for r in runs:
    status = r.data.tags.get("review-status", "—")
    print(round(r.data.metrics.get("f1_score", 0), 2), '|', status, '|', r.data.tags.get("mlflow.runName"))
EOF
```

## Key Takeaways

### 🔍 `client.search_runs()` — querying runs programmatically
`search_runs(experiment_id, filter_string, order_by)` returns a list of runs matching an optional filter. Sorting by `metrics.f1_score DESC` immediately surfaces the best run without manually scanning the UI. For larger experiment histories this is far faster than clicking through pages.

### 🏷️ Run-level tags for workflow state
Tags on runs are the standard MLflow mechanism for encoding workflow state — review status, deployment readiness, data quality flags. Unlike metrics and parameters (which are immutable after logging), tags can be added, updated, or deleted at any time via `client.set_tag()` or the UI, making them ideal for post-hoc triage and annotation.

### 🎯 Shortlisting logic — highest f1 above threshold
The task requires tagging only the single highest-scoring run above 0.85, not all runs above the threshold. Sorting by metric descending and taking the first result is the correct pattern — avoid tagging all candidates above the bar, which would produce multiple `shortlisted` runs and fail the check.

### 🚫 Runs with no tag — intentional absence
The 0.75 ≤ f1 ≤ 0.85 band must carry no `review-status` tag. This is an important detail: adding a neutral tag like `review-status=pending` to those runs would also fail the check. When a condition requires the absence of a tag, the correct action is to leave those runs untouched.
