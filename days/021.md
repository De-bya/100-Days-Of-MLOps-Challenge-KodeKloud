# Day 021: Log an ML Experiment to MLflow

**Category**: ML Experimentation | **Difficulty**: 🟢 Beginner

---

## Problem

A xFusionCorp Industries data scientist needs a training run recorded in MLflow so the team has a baseline record on the tracking dashboard. The non-MLflow scaffolding has already been written at `/root/code/log_experiment.py`; the MLflow logging calls are left as TODO blocks. Complete the script so that every element of the run is captured by the MLflow tracking server.

1. The MLflow tracking server is already running on port `5000`. The **MLflow UI** button at the top of the lab can be opened to view the dashboard; the `Default` experiment is present on first load.
2. `/root/code/log_experiment.py` can be opened in the VS Code editor. The script prepares a `params` dictionary, fits a trivial sklearn model, and advertises a pair of synthetic evaluation scores (`accuracy` and `f1`). Three blocks marked `# TODO` inside the `mlflow.start_run()` context are the only edits required.
3. Execute the script once (`python3 /root/code/log_experiment.py`) after the TODOs are completed. The end state must include:
   - Exactly one new run in the `Default` experiment.
   - Every hyperparameter in the `params` dict (`n_estimators=100`, `max_depth=5`, `random_state=42`) recorded as a run parameter.
   - Both advertised scores (`accuracy`, `f1_score`) recorded as run metrics.
   - The sklearn model captured as an MLflow model artefact on the run.

## Solution

### Step 1: Inspect the existing script

```bash
cat /root/code/log_experiment.py
```

The script already sets the tracking URI, builds the `params` dict, fits a `DummyClassifier`, and defines `accuracy = 0.92` and `f1 = 0.89`. Only the three TODO blocks inside `with mlflow.start_run():` need to be filled in.

### Step 2: Edit the script

```bash
nano /root/code/log_experiment.py
```

Replace the three TODO comments with the logging calls:

```python
with mlflow.start_run():
    # TODO 1: log every entry in `params` as an MLflow parameter
    mlflow.log_params(params)

    # TODO 2: log accuracy and f1 as MLflow metrics
    mlflow.log_metric("accuracy", accuracy)
    mlflow.log_metric("f1_score", f1)

    # TODO 3: log the trained model as an MLflow sklearn model artefact
    mlflow.sklearn.log_model(model, "model")

    print(f"accuracy={accuracy}, f1_score={f1}")
```

Save with `Ctrl+O` → `Enter` → `Ctrl+X`.

### Step 3: Run the script

```bash
python3 /root/code/log_experiment.py
```

Expected output:

```
accuracy=0.92, f1_score=0.89
🏃 View run likeable-perch-911 at: http://localhost:5000/#/experiments/0/runs/372f9705f6d844c499f0952e5e2cf250
🧪 View experiment at: http://localhost:5000/#/experiments/0
```

The run is now visible in the MLflow UI under the `Default` experiment with all three parameters, both metrics, and the serialised model artefact.

## Key Takeaways

### 🏃 `mlflow.start_run()` — the run context manager
Every MLflow logging call must happen inside a `with mlflow.start_run():` block. This context manager opens a new run, assigns it a unique run ID, and automatically closes and finalises the run when the block exits — even if an exception is raised. Anything logged outside the block is silently ignored.

### 🔢 `mlflow.log_params()` vs `mlflow.log_param()`
`mlflow.log_param(key, value)` logs a single parameter. `mlflow.log_params(dict)` logs an entire dictionary in one call — cleaner when you already have a `params` dict in your training script. Parameters are intended for configuration values that do not change during a run (hyperparameters, dataset paths, random seeds).

### 📈 `mlflow.log_metric()` — tracking scalar scores
`mlflow.log_metric(key, value)` records a numeric result. Metrics support an optional `step` argument for logging values over training iterations (e.g. loss per epoch). Without `step`, the value is recorded at step 0 — fine for end-of-training summary metrics like accuracy and F1.

### 📦 `mlflow.sklearn.log_model()` — serialising the model
`mlflow.sklearn.log_model(model, name)` serialises the fitted sklearn estimator and stores it as a named artefact directory on the run. MLflow saves the model binary alongside a `MLmodel` metadata file that records the flavour, Python environment, and input/output schema — enough information to reload the model later with `mlflow.sklearn.load_model()` or serve it via `mlflow models serve`.

### 🔗 `mlflow.set_tracking_uri()` — pointing at the server
By default MLflow logs to a local `mlruns/` directory. Calling `mlflow.set_tracking_uri("http://localhost:5000")` at the top of the script redirects all logging to the running tracking server instead, so runs appear in the shared UI and are persisted in the SQLite backend rather than scattered across local directories.
