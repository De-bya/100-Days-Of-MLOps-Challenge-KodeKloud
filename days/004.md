# Day 004: ML Project Structure & Layout Conventions

**Category**: ML Project Organization | **Difficulty**: 🟢 Beginner

---

## 📋 Problem

A colleague has started a new ML project at `/root/code/fraud-detection/`, but the layout does not match the xFusionCorp Industries standard. Bring the project in line with the team's conventions.

The final layout must match this tree exactly:

```
fraud-detection/
├── data/
│   ├── raw/
│   └── processed/
├── models/
├── notebooks/
├── src/
│   ├── __init__.py
│   ├── data/
│   │   └── __init__.py
│   ├── features/
│   │   └── __init__.py
│   ├── models/
│   │   └── __init__.py
│   └── utils/
│       └── __init__.py
├── tests/
├── configs/
├── requirements.txt
└── README.md
```

Requirements:
1. Every subdirectory under `src/` must contain an `__init__.py` file so Python recognises it as a package.
2. `requirements.txt` must list: `scikit-learn`, `pandas`, `numpy`, and `mlflow` (one per line).
3. `README.md` must begin with the heading `# fraud-detection`.

Bugs found in the existing project:
- `data/raw/` and `data/processed/` were missing
- `src/feature/` was misspelled → should be `src/features/`
- `src/util/` was misspelled → should be `src/utils/`
- `src/__init__.py` was missing
- `tests/` and `configs/` directories were missing
- `requirements.txt` had `sklearn` instead of `scikit-learn` and was missing `mlflow`
- `README.md` started with `# Fraud` instead of `# fraud-detection`

---

## ✅ Solution

### Step 1: Inspect the existing project

```bash
ls -R /root/code/fraud-detection/
```

### Step 2: Fix `requirements.txt`

```bash
nano /root/code/fraud-detection/requirements.txt
```

Set the contents to:
```
scikit-learn
pandas
numpy
mlflow
```

### Step 3: Fix `README.md`

```bash
nano /root/code/fraud-detection/README.md
```

Make sure the first line is exactly:
```
# fraud-detection
```

### Step 4: Create missing directories and fix structure

```bash
cd /root/code/fraud-detection

# Create missing directories
mkdir -p data/raw data/processed tests configs

# Fix misspelled folder names
mv src/feature src/features
mv src/util src/utils

# Add missing __init__.py at src root
touch src/__init__.py
```

### Step 5: Verify

```bash
ls -R /root/code/fraud-detection/
```

## 💡 Key Takeaways

### 🗂️ Why ML projects need a standard structure
In MLOps, a consistent project layout is not just aesthetic — it's functional. When every project follows the same structure, team members can navigate any codebase instantly, CI/CD pipelines know where to find tests and configs, and tools like MLflow and DVC have predictable paths to hook into. Agreeing on a layout is one of the first things a mature ML team does.

### 🐍 `__init__.py` — Making directories into Python packages
An `__init__.py` file (even if empty) tells Python: "treat this directory as a package." Without it, you cannot do `from src.features import my_function` — Python simply won't recognize `src/features` as importable. Every subdirectory under `src/` needs one. This is a very common source of `ModuleNotFoundError` in ML projects.

### 📁 `data/raw/` vs `data/processed/`
A core principle in data engineering and MLOps is to **never overwrite raw data**. Raw data goes into `data/raw/` and is treated as read-only. Any cleaning, transformation, or feature engineering outputs go into `data/processed/`. This separation makes pipelines reproducible — you can always rerun processing from the original source.

### 🧪 `tests/` and `configs/`
- `tests/` holds unit and integration tests for your ML code. Testing is not optional in production ML — you test data pipelines, feature transformations, and model interfaces just like any software.
- `configs/` stores configuration files (YAML, JSON) for experiments, model hyperparameters, and pipeline settings. Keeping config out of code makes it easy to run experiments without touching source files.
