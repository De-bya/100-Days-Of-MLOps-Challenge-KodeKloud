# Day 025: Register, Version, and Manage Model Lifecycle

**Category**: ML Experimentation | **Difficulty**: 🟡 Intermediate

---

## Problem

The xFusionCorp Industries ML platform team needs two trained candidates promoted through the MLflow Model Registry so the ops side can track which model version is serving production traffic. Both runs already exist in the `fraud-detection` experiment. Register both as versions of a new `fraud-detector` model, add a model-level description, and assign `challenger` and `champion` aliases.

1. The MLflow tracking server is already running on port `5000`. Two runs are pre-populated in the `fraud-detection` experiment:
   - `baseline` run (`n_estimators=100`, `max_depth=5`, `f1_score=0.80`)
   - `improved` run (`n_estimators=200`, `max_depth=10`, `f1_score=0.89`)
2. Register both runs sequentially (baseline first, improved second) as versions of a registered model named `fraud-detector`.
3. The registered model must carry a non-empty description referencing the word `fraud`.
4. Version 1 (`baseline`) carries the alias `challenger`.
5. Version 2 (`improved`) carries the alias `champion`.

---

## Solution

### Step 1: Identify the run IDs

```bash
python3 << 'EOF'
import mlflow
mlflow.set_tracking_uri("http://localhost:5000")
client = mlflow.tracking.MlflowClient()

exp = client.get_experiment_by_name("fraud-detection")
runs = client.search_runs(exp.experiment_id, order_by=["metrics.f1_score ASC"])
for r in runs:
    print(r.info.run_id, '|', r.data.tags.get("mlflow.runName"), '|', r.data.metrics)
EOF
```

Output:

```
e36a1c4d978948b6a62ee5e9068405cf | baseline | {'accuracy': 0.82, 'f1_score': 0.8}
7b7bc1c8c94f4f389968eebe48ac9815 | improved | {'accuracy': 0.91, 'f1_score': 0.89}
```

### Step 2: Register both runs, add description and aliases

```bash
python3 << 'EOF'
import mlflow
mlflow.set_tracking_uri("http://localhost:5000")
client = mlflow.tracking.MlflowClient()

# Register baseline first (becomes v1), improved second (becomes v2)
v1 = mlflow.register_model("runs:/e36a1c4d978948b6a62ee5e9068405cf/model", "fraud-detector")
v2 = mlflow.register_model("runs:/7b7bc1c8c94f4f389968eebe48ac9815/model", "fraud-detector")

# Add model-level description
client.update_registered_model(
    "fraud-detector",
    description="Fraud detection model for xFusionCorp transactions"
)

# Assign aliases
client.set_registered_model_alias("fraud-detector", "challenger", v1.version)
client.set_registered_model_alias("fraud-detector", "champion", v2.version)

print(f"v{v1.version}=challenger, v{v2.version}=champion")
EOF
```

Output:

```
v1=challenger, v2=champion
```

The `fraud-detector` model now appears in the Model Registry with two versions, a description, and both aliases assigned correctly.

## Key Takeaways

### 📋 The MLflow Model Registry
The Model Registry is a centralised catalogue of named, versioned models. It sits on top of the experiment tracking layer — runs log model artefacts, and the registry promotes selected artefacts into named versions that ops teams can reference by name and alias rather than by run ID.

### 🔢 `mlflow.register_model()` — promoting a run artefact
`mlflow.register_model("runs:/<run_id>/model", "model-name")` copies the artefact from the run into the registry and assigns it the next available version number. Registration order determines version numbers — baseline registered first becomes v1, improved second becomes v2.

### 🏷️ Aliases — stable references for deployment
Aliases (`challenger`, `champion`) are mutable pointers to specific versions. Downstream serving code can load a model by alias (`models:/fraud-detector@champion`) instead of by version number, so promoting a new champion only requires updating the alias — not changing any serving code.

### 📝 `client.update_registered_model()` — model-level metadata
Description and tags set on the registered model itself (not on a version) apply across all versions. This is the right place for project-level context — what the model does, who owns it, which team approved it.

### 🔄 Aliases vs the legacy Staging/Production stages
MLflow 3.x deprecates the old `Staging` / `Production` / `Archived` stage transitions in favour of aliases. Aliases are more flexible — you can define any workflow vocabulary (`challenger`, `champion`, `shadow`, `canary`) without being locked into a fixed three-stage lifecycle.
