# Day 029: Configure MLflow with Remote Tracking Server and Artifact Store

**Category**: ML Experimentation | **Difficulty**: 🟡 Intermediate

---

## Problem

The xFusionCorp Industries ML platform team rolled out a shared MLflow deployment backed by a production-style tracking store (PostgreSQL) and a production-style artifact store (SeaweedFS, S3-compatible). The MLflow server is running and metadata writes to PostgreSQL succeed, but a teammate noticed that no artifact ever lands in the `mlflow-artifacts` SeaweedFS bucket — the connection from MLflow to SeaweedFS is not wired correctly. Diagnose what is broken in the MLflow server's startup configuration, fix it, and re-run the pre-staged smoke-test so one round trip lands metadata in PostgreSQL **and** the model artifact in SeaweedFS.

1. PostgreSQL (`mlflow-db`, port `5432`, db `mlflow`, user `mlflow`/`mlflow123`) and SeaweedFS (S3 API on `8333`, Filer UI on `8888`, bucket `mlflow-artifacts`, credentials `weedadmin`/`weedadmin123`) are already running.
2. The MLflow tracking server is running on port `5000`, launched by `/root/code/start-mlflow.sh`.
3. Inspect `/root/code/start-mlflow.sh` and reconcile its environment with the SeaweedFS endpoint, then restart the server with `bash /root/code/restart-mlflow.sh`.
4. Re-run the smoke-test with `python3 /root/code/log_test_run.py`.
5. The end state must include a successful run in the `test_remote` experiment, the model artifact visible in the SeaweedFS Filer, and the run's metadata in the PostgreSQL `mlflow` database.

---

## Solution

### Step 1: Inspect the startup script

The script sets `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY` but never points boto3 at the local SeaweedFS endpoint. Without `MLFLOW_S3_ENDPOINT_URL`, MLflow's S3 client defaults to real AWS S3 — so every artifact upload silently fails to reach SeaweedFS, even though the credentials are otherwise correct.

### Step 2: Add the missing endpoint variable

Open `/root/code/start-mlflow.sh` in VS Code and add one line:

```bash
export MLFLOW_S3_ENDPOINT_URL=http://localhost:8333
```

### Step 3: Restart the server and re-run the smoke-test

```bash
bash /root/code/restart-mlflow.sh
python3 /root/code/log_test_run.py
```

The run now completes end to end — metadata lands in PostgreSQL and the model artifact uploads successfully to the SeaweedFS `mlflow-artifacts` bucket.

---

## Key Takeaways

### 🔌 `MLFLOW_S3_ENDPOINT_URL` — redirecting boto3 to a custom S3 endpoint
MLflow uses boto3 under the hood for any `s3://` artifact store. boto3 defaults to AWS's real S3 endpoints unless explicitly told otherwise. Setting `MLFLOW_S3_ENDPOINT_URL` redirects every S3 API call to a self-hosted, S3-compatible store — SeaweedFS, MinIO, or any other implementation — without changing a single line of application code.

### 🔑 Credentials being correct isn't enough
`AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY` authenticate the request, but they say nothing about *where* the request goes. A fully correct credential pair against the wrong endpoint fails just as silently as a missing one — the error surfaces as a connection or upload failure, not an auth failure, which is why it's easy to overlook during diagnosis.

### 🧩 Two independent backends, two independent failure modes
A production MLflow deployment separates the **backend store** (run metadata — PostgreSQL here) from the **artifact store** (large binary files — SeaweedFS here). One can work perfectly while the other is silently broken, because they're configured by entirely separate flags and environment variables. Always verify both halves independently when diagnosing a "partially working" MLflow server.
