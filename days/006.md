# Day 006: Code Quality with Ruff and Black

**Category**: Code Quality / Linting | **Difficulty**: 🟢 Beginner 

---

## Problem

The xFusionCorp Industries ML team enforces code quality with `ruff` and `black` on every pull request. The project at `/root/code/fraud-detection/` currently fails both tools. Make it pass them.

Given context:
1. The project contains a `pyproject.toml` and sample sources under `src/`
2. The corrected project must meet the following requirements:
   - `ruff` and `black` are both configured with a line length of `120`
   - `ruff` lint rule selection includes `E`, `F`, `W`, and `I`, and is declared under `[tool.ruff.lint]` — the schema required by ruff 0.1 and later
   - Running `ruff check src/` exits with status 0
   - Running `black --check src/` exits with status 0
3. Review the existing configuration and source files, and correct everything that prevents the two commands above from exiting cleanly

Bugs found in the existing `pyproject.toml`:
- `line-length = 88` under `[tool.ruff]` → should be `120`
- `line-length = 100` under `[tool.black]` → should be `120`
- `select` was declared directly under `[tool.ruff]` → must be under `[tool.ruff.lint]`
- `select` was missing rules `W` and `I`

---

## Solution

### Step 1: Inspect the broken config

```bash
cat /root/code/fraud-detection/pyproject.toml
```

### Step 2: Fix pyproject.toml

```bash
cat > /root/code/fraud-detection/pyproject.toml << 'EOF'
[project]
name = "fraud-detection"
version = "0.1.0"

[tool.ruff]
line-length = 120

[tool.ruff.lint]
select = ["E", "F", "W", "I"]

[tool.black]
line-length = 120
EOF
```

### Step 3: Auto-fix and format source files

```bash
cd /root/code/fraud-detection
ruff check src/ --fix
black src/
```

### Step 4: Verify both pass

```bash
ruff check src/
black --check src/
```

Both should exit cleanly with no errors.

## Key Takeaways

### 🔍 What is a Linter?
A linter is a tool that statically analyzes your code — without running it — to catch bugs, style issues, and bad practices. In ML projects, linting is critical because data scientists often write exploratory code that later ends up in production. A linter enforces a baseline of quality before code ever gets reviewed or deployed.

### ⚡ Ruff — the fast Python linter
`ruff` is a modern Python linter written in Rust, designed to replace tools like `flake8`, `isort`, and `pyupgrade` all at once. It is 10-100x faster than traditional linters and is becoming the standard in modern Python and MLOps projects. Ruff uses rule codes to define what it checks:

| Rule set | What it catches |
|----------|----------------|
| `E` | PEP8 style errors |
| `F` | Pyflakes errors (unused imports, undefined names) |
| `W` | PEP8 warnings |
| `I` | Import sorting (replaces isort) |

### 🖤 Black — the uncompromising formatter
`black` is an opinionated code formatter — it reformats your entire codebase to a single consistent style with zero configuration debates. Unlike a linter that just flags issues, black actually rewrites your files. The `--check` flag runs black without making changes, returning exit code 1 if any file would be reformatted (used in CI).

### 📏 Line length — why 120?
The default line length for black is 88 and for ruff is 88 too. Teams often bump this to 120 for ML projects because data science code tends to have long variable names, chained pandas operations, and verbose model configs. The important thing is that **both tools agree on the same value** — mismatched line lengths cause one tool to flag lines the other already accepted.

### 🗂️ `pyproject.toml` — the modern Python config file
`pyproject.toml` is the standard single configuration file for modern Python projects, replacing `setup.py`, `setup.cfg`, `.flake8`, and other scattered config files. All tools — ruff, black, pytest, mypy — can read their config from the same file under their own `[tool.X]` section. This is why the section name matters exactly: `[tool.ruff.lint]` is not the same as `[tool.ruff]`.

### 🔄 Linting in MLOps CI/CD
In a real MLOps pipeline, linting runs automatically on every pull request. If `ruff check` or `black --check` exits with a non-zero status, the PR is blocked. This keeps the codebase clean without relying on manual reviews for style issues, freeing reviewers to focus on logic and model quality.
