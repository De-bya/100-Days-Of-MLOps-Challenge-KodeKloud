# Day 003: Dependency Management with `uv` and Lockfiles

**Category**: Dependency Management | **Difficulty**: 🟢 Beginner

---

## 📋 Problem

The xFusionCorp Industries ML team uses `uv` and lockfiles to keep Python dependencies reproducible across machines. A teammate has left behind a `requirements.in` specification that does not match the team's standard. Correct it and compile it into a pinned lockfile.

Given context:
1. A high-level dependency specification exists at `/root/code/fraud-detection/requirements.in`. `uv` is already installed.
2. The corrected specification must meet the following requirements:
   - it lists exactly these four top-level packages: `scikit-learn`, `mlflow`, `pandas`, and `numpy`
   - every package carries a version constraint that `uv` can actually satisfy against PyPI
3. Review the existing `requirements.in`, and correct everything that does not match the requirements above.
4. From the project directory, compile the corrected specification into a pinned lockfile:
```bash
uv pip compile requirements.in -o requirements.txt
```
5. The resulting `requirements.txt` must pin each of the four top-level packages to an exact version using `==`, and must also include the transitive dependencies that `uv` resolved.

The actual bugs found in the file:
- `sklearn` → wrong package name on PyPI, the correct name is `scikit-learn`
- `mlflow>=100.0` → version 100.0 does not exist, making it unresolvable
- `pandas` → missing entirely from the specification

---

## ✅ Solution

### Step 1: Read the broken file

```bash
cat /root/code/fraud-detection/requirements.in
```

Output revealed:
```
# Fraud detection project dependencies
sklearn
mlflow>=100.0
numpy
```

### Step 2: Fix the file

```bash
nano /root/code/fraud-detection/requirements.in
```

Edit it to look like this:

```
# Fraud detection project dependencies
scikit-learn
mlflow
pandas
numpy
```

Save with `Ctrl+O` → `Enter` → `Ctrl+X`

### Step 3: Compile into a pinned lockfile

```bash
cd /root/code/fraud-detection
uv pip compile requirements.in -o requirements.txt
```

### Step 4: Verify

```bash
cat requirements.txt
```

You should see all 4 packages pinned with `==` plus all their transitive dependencies.

## 💡 Key Takeaways

### 📦 PyPI — Python Package Index
PyPI (pypi.org) is the official repository where Python packages are published and downloaded from. When you run `pip install numpy` or `uv pip compile`, the tool reaches out to PyPI to find and download the package. Package names on PyPI are canonical — `scikit-learn` is the correct name, not `sklearn` (`sklearn` is just the import name inside your code). 

### ⚡ `uv` — Fast Python Package Manager
`uv` is a modern, extremely fast Python package and project manager written in Rust, designed as a drop-in replacement for `pip` and `pip-tools`. It resolves dependencies much faster than pip and is becoming the standard in production ML workflows. Key commands:
```bash
uv pip install <package>        # install a package
uv pip compile requirements.in -o requirements.txt  # compile a lockfile
uv pip sync requirements.txt    # install exact versions from lockfile
```

### 📄 `requirements.in` — High-level Specification
This is where you list only your **direct** dependencies — the packages your project actually uses — optionally with loose version constraints (e.g. `pandas`, `numpy>=1.24`). It is human-maintained and kept minimal. 

### 🔒 `requirements.txt` — Pinned Lockfile
This is the **compiled output** generated from `requirements.in`. It pins every package (direct and transitive) to an exact version using `==`. It guarantees that every machine, every CI run, and every teammate installs the exact same environment. You never edit this file by hand — you always regenerate it with `uv pip compile`.


### 🌊 Transitive Dependencies
When you declare `mlflow` as a dependency, mlflow itself depends on dozens of other packages. Those are called **transitive dependencies**. `uv pip compile` resolves the full tree and pins everything — so you never get a surprise version mismatch from a package you didn't even know you were using.
