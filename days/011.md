# Day 011: Track a Dataset with DVC

**Category**: Data Version Control | **Difficulty**: 🟢 Beginner

---

## Problem

A teammate has added the transactions dataset to the xFusionCorp Industries fraud-detection repository, but it was committed directly to Git instead of being tracked with DVC. Bring the repository in line with the team standard — every dataset under `data/` must be tracked by DVC, not by Git.

1. A project exists at `/root/code/fraud-detection/` with DVC already initialised. The dataset `data/raw/transactions.csv` is currently tracked by Git, and the team standard requires DVC to own it instead.
2. Stop Git from tracking the dataset without deleting it from disk.
3. Track the same dataset with DVC so a `.dvc` pointer file is produced and `data/raw/.gitignore` excludes the dataset itself.
4. Stage the new `.dvc` pointer and the new `.gitignore`, then record a Git commit with the message `Track transactions dataset with DVC`.

---

## Solution

### Step 1: Navigate to the repository

```bash
cd /root/code/fraud-detection
```

### Step 2: Remove the file from Git tracking (without deleting it from disk)

```bash
git rm --cached data/raw/transactions.csv
```

### Step 3: Track it with DVC

```bash
dvc add data/raw/transactions.csv
```

### Step 4: Stage the DVC-produced files and commit

```bash
git add data/raw/.gitignore data/raw/transactions.csv.dvc
git commit -m "Track transactions dataset with DVC"
```

---

## Key Takeaways

### 📦 What does "tracked by Git" actually mean?
When a file is tracked by Git, it gets committed to your repository history and uploaded to GitHub on every `git push`. This means every teammate who runs `git clone` downloads that file. For a 500MB dataset that is a serious problem — GitHub enforces a 100MB file size limit and will outright reject large files. Data files do not belong in Git.

### 🗄️ What does "DVC owns the file" mean?
When you run `dvc add` on a file, DVC takes ownership of it in two ways:
- It copies the file into DVC's local cache (`.dvc/cache/`) on your machine
- It creates a tiny `.dvc` pointer file that Git tracks instead of the real data

The pointer file looks like this:
```yaml
outs:
- md5: a1b2c3d4e5f6...
  size: 524288000
  path: transactions.csv
```

That checksum is all Git ever stores. When a teammate needs the actual data they run `dvc pull` and DVC fetches it from a remote storage backend (S3, GCS, Azure, etc.) using that checksum.

### 🧠 The full picture

```
BEFORE:                            AFTER:
GitHub stores:                     GitHub stores:
  data/raw/transactions.csv   →      data/raw/transactions.csv.dvc  (5 lines)
  (the actual 500MB file!)           data/raw/.gitignore

                                   DVC remote (S3/local) stores:
                                     data/raw/transactions.csv  (real file)
```

Git stays lightweight and fast. DVC handles the heavy data. Teammates get the code instantly via `git clone` and pull the data separately via `dvc pull`.

### 🔄 The DVC data workflow going forward
```
Add/update data   →  dvc add data/raw/file.csv
                  →  git add data/raw/file.csv.dvc
                  →  git commit -m "Update dataset"
                  →  git push
                  →  dvc push   (sends data to remote storage)

Teammate syncs    →  git pull   (gets the pointer)
                  →  dvc pull   (gets the actual data)
```
