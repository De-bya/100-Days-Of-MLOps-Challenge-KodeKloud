# Day 002: JupyterLab Server Configuration & Debugging

**Category**: ML Environment | **Difficulty**: 🟢 Beginner

---

## 📋 Problem

A teammate has configured a JupyterLab server for the xFusionCorp Industries data science team, but the server does not behave correctly. The task is to inspect the configuration, diagnose the issues, and start the server.

Given context:
1. JupyterLab is already installed in the virtual environment at `/root/code/ml-env/`. The team's configuration file is at `/root/code/jupyter_lab_config.py` and is visible in the file explorer.
2. When JupyterLab is started, the **Jupyter UI** button at the top of the lab must open the notebook interface.
3. For this to work, the running server must meet the following requirements:
   - it listens on port `8888`
   - it binds on `0.0.0.0` (the lab proxy cannot reach a server that is only bound on `127.0.0.1`)
   - the notebook root directory is `/root/notebooks/`, and that directory exists on disk
4. Open the configuration file, identify every setting that prevents the requirements above from being met, and correct it. Create any missing directories.
5. Start JupyterLab from the virtual environment using the corrected configuration:
```bash
source /root/code/ml-env/bin/activate
jupyter lab --config=/root/code/jupyter_lab_config.py --allow-root --no-browser &
```

---

## ✅ Solution

### Step 1: Read the current config

```bash
cat /root/code/jupyter_lab_config.py
```

### Step 2: Fix the broken setting

```bash
nano /root/code/jupyter_lab_config.py
```

Find and fix these 3 lines so they look exactly like this:

```python
c.ServerApp.port = 8888
c.ServerApp.ip = '0.0.0.0'
c.ServerApp.root_dir = '/root/notebooks/'
```

Save with `Ctrl+O` → `Enter` → `Ctrl+X`

### Step 3: Create the notebooks directory if missing

```bash
mkdir -p /root/notebooks
```

### Step 4: Activate the venv and start JupyterLab

```bash
source /root/code/ml-env/bin/activate
jupyter lab --config=/root/code/jupyter_lab_config.py --allow-root --no-browser &
```

## 💡 Key Takeaways

### 🌐 Network binding: `0.0.0.0` vs `127.0.0.1`
This is one of the most common "why can't I reach my server?" problems in MLOps. When a server binds to `127.0.0.1` (localhost), it only accepts connections from the same machine. When it binds to `0.0.0.0`, it listens on all network interfaces and becomes reachable from outside — essential when your server sits behind a proxy, inside a container, or on a remote machine. In production ML systems, your model servers, experiment trackers, and notebook servers all need to be reachable by other services, so understanding this distinction is critical.

### 📄 Config files are the source of truth
In MLOps, almost every tool — JupyterLab, MLflow, Airflow, Kubeflow — is driven by a config file. Learning to read, understand, and debug these files is a core skill. Always inspect the config first before assuming the tool itself is broken.
