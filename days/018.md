# Day 018: Version Datasets and Models Across Git Branches

**Category**: ML Experimentation | **Difficulty**: 🟡 Intermediate

---

## Problem

The xFusionCorp Industries ML team keeps different dataset and model versions on different Git branches so that the team can roll between versions cleanly. Tag the current state as `v1.0`, produce a `v2-improved` branch based on a newer dataset, and confirm that switching back restores the original data.

1. A project exists at `/root/code/fraud-detection/` with a working DVC pipeline and the baseline `data/raw/transactions.csv` already tracked.
2. An improved dataset has been pre-staged at `/root/code/fraud-detection/data/raw/transactions_v2.csv`. Do not delete this file.
3. On the `main` branch, tag the current state as `v1.0`.
4. Create a new branch named `v2-improved`. Replace the tracked dataset with the contents of the v2 file, re-track it with DVC, re-run the pipeline, and commit the changes.
5. Switch back to the `main` branch and use `dvc checkout` to restore the v1 dataset on disk. The restored content must match the hash recorded by the `v1.0` tag.

---

## Solution

### Step 1: Verify the baseline state

```bash
cd /root/code/fraud-detection
git status
dvc status
```

Both should report a clean working state before proceeding.

### Step 2: Tag the current state as `v1.0`

```bash
git tag v1.0
```

### Step 3: Create the `v2-improved` branch

```bash
git checkout -b v2-improved
```

### Step 4: Replace the tracked dataset with the v2 file

```bash
cp data/raw/transactions_v2.csv data/raw/transactions.csv
```

### Step 5: Re-track the new dataset with DVC

```bash
dvc add data/raw/transactions.csv
```

DVC updates `data/raw/transactions.csv.dvc` with the new file hash.

### Step 6: Re-run the pipeline

```bash
dvc repro
```

Only the stages dependent on `transactions.csv` rerun (`process_data`, `split_data`). DVC updates `dvc.lock` accordingly.

### Step 7: Commit the changes on `v2-improved`

```bash
git add data/raw/transactions.csv.dvc dvc.lock
git commit -m "Use v2 dataset and re-run pipeline"
```

### Step 8: Switch back to `main` and restore v1 data

```bash
git checkout main
dvc checkout
```

`dvc checkout` restores all files tracked by `main`'s `.dvc` files back to their v1 state on disk.

### Step 9: Verify the restore

```bash
dvc status
```

Output should confirm:

```
Data and pipelines are up to date.
```

All four files are restored to their original v1 hashes:

```
M       data/processed/clean_transactions.csv
M       data/processed/test.csv
M       data/processed/train.csv
M       data/raw/transactions.csv
```

## Key Takeaways

### 🏷️ `git tag` — snapshotting a version
A Git tag marks a specific commit as a named reference point. Tagging `v1.0` on `main` before branching means you can always return to the exact commit — and therefore the exact DVC-tracked data — that corresponds to your baseline model version.

### 🌿 Branching for dataset versions
Each Git branch holds its own `.dvc` files and `dvc.lock`, which point to different file hashes in DVC cache. This means switching Git branches effectively switches your dataset and pipeline outputs — as long as you follow up with `dvc checkout`.

### 🔄 `dvc checkout` — syncing disk with the branch
After `git checkout`, your `.dvc` files update to the branch's version, but the actual data files on disk do not change automatically. Running `dvc checkout` reads the hashes from the current branch's `.dvc` files and restores the correct data files from the local DVC cache.

### 🔁 `dvc repro` — only reruns what changed
`dvc repro` is stage-aware. When the tracked dataset changes, only the downstream stages rerun. In this pipeline, `process_data` and `split_data` reran while unaffected stages were skipped — keeping experimentation fast.

### 📦 DVC cache — the backbone of data versioning
DVC stores file content by hash in `.dvc/cache`. Both branches share the same cache, so switching between `main` and `v2-improved` never duplicates data on disk — DVC simply hard-links or copies the right cached version when you run `dvc checkout`.
