# Day 015: Manage Hyperparameters with DVC Params

**Category**: Data Version Control / ML Pipelines | **Difficulty**: 🟢 Beginner

## Problem

The xFusionCorp Industries ML team manages model hyperparameters through `params.yaml` so experiments can vary without code changes. The fraud-detection project's `train` stage already wires `params.yaml` for `n_estimators`, but `dvc repro` currently fails. Correct the parameter wiring and demonstrate that DVC re-runs the train stage when the parameter changes.

Given context:
1. A project exists at `/root/code/fraud-detection/` with a three-stage DVC pipeline (`process_data`, `split_data`, `train`) and a `params.yaml` already in place. Do not modify the Python files.
2. The `train` stage in `dvc.yaml` references the `n_estimators` parameter. Every name listed under `params:` must resolve to a key in `params.yaml`.
3. Review `params.yaml`, correct whatever prevents `dvc repro` from completing, and run the full pipeline.
4. Demonstrate that DVC tracks parameter changes by updating `n_estimators` to `200`, running `dvc repro` again — only the `train` stage should re-execute, the new value must be recorded in `dvc.lock`, and `models/model.pkl` must be regenerated.

## Solution

### Step 1: Inspect the files

```bash
cat /root/code/fraud-detection/dvc.yaml
cat /root/code/fraud-detection/params.yaml
```

### Step 2: Fix the typo in params.yaml

```bash
nano /root/code/fraud-detection/params.yaml
```

Change to:
```yaml
n_estimators: 100
```

Save with `Ctrl+O` → `Enter` → `Ctrl+X`

### Step 3: Run the full pipeline

```bash
cd /root/code/fraud-detection
dvc repro
```

All three stages run in order.

### Step 4: Change the parameter and re-run

```bash
nano /root/code/fraud-detection/params.yaml
```

Change to:
```yaml
n_estimators: 200
```

```bash
dvc repro
```

Only the `train` stage reruns — `process_data` and `split_data` are skipped.

### Step 5: Verify

```bash
dvc status        # should say: Data and pipelines are up to date
cat dvc.lock      # n_estimators: 200 recorded under train stage
ls models/        # model.pkl regenerated
```

## Key Takeaways

### 🎛️ What is `params.yaml`?
`params.yaml` is a plain YAML file that holds all the hyperparameters and configuration values for your ML pipeline. Instead of hardcoding values like `n_estimators=100` inside your Python scripts, you read them from this file. This means you can change hyperparameters without touching any code — just edit `params.yaml` and rerun.

### 🔗 How DVC wires params to stages
In `dvc.yaml`, the `params:` field under a stage tells DVC which keys from `params.yaml` that stage depends on:

```yaml
train:
  cmd: python src/models/train.py
  params:
    - n_estimators      # DVC watches this key in params.yaml
```

If the value of `n_estimators` changes, DVC marks the `train` stage as stale and reruns it on the next `dvc repro`. If it hasn't changed, the stage is skipped — saving time.

### ⚡ Smart re-execution — only what changed
When you changed `n_estimators` from 100 to 200 and ran `dvc repro` again:
- `process_data` → skipped (input data didn't change)
- `split_data` → skipped (clean_transactions.csv didn't change)
- `train` → **reran** (n_estimators changed)

This is DVC's core value: it only reruns the stages that are actually affected by a change. In a real pipeline with hours of preprocessing, this saves enormous amounts of time.

### 📋 `dvc.lock` — the experiment snapshot
`dvc.lock` is automatically updated after every `dvc repro`. It records the exact state of the pipeline at that moment — the MD5 hash of every input file, output file, and parameter value. This is what makes ML experiments reproducible: given a `dvc.lock`, you can always reconstruct the exact environment that produced a model.

```yaml
# dvc.lock records the parameter value used
train:
  params:
    params.yaml:
      n_estimators: 200    # exact value used in this run
```

### 🧪 This is the foundation of experiment tracking
Changing `params.yaml` + `dvc repro` + committing `dvc.lock` to Git is the simplest form of ML experiment tracking. Each Git commit captures a full snapshot of: which code ran, which data was used, which hyperparameters were set, and what model was produced. Tools like MLflow and DVC Experiments build on this foundation to make it easier to compare runs at scale.
