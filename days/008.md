# Day 008: Configure Pre-Commit Hooks for ML Repository

**Category**: Code Quality / Git Automation | **Difficulty**: 🟢 Beginner 

---

## Problem

The xFusionCorp Industries ML team enforces code quality on every commit via `pre-commit`. A draft `.pre-commit-config.yaml` exists in the git repository at `/root/code/fraud-detection/`, but it does not match the team's standard and `pre-commit run --all-files` fails against it. Correct the configuration.

The corrected configuration must declare five hooks:
- `trailing-whitespace`, `end-of-file-fixer`, and `check-yaml` — sourced from `pre-commit/pre-commit-hooks`, pinned to a current release
- `ruff` — sourced from the `astral-sh/ruff-pre-commit` repository, pinned to a current release
- `black` — sourced from the `psf/black-pre-commit-mirror` repository, pinned to a current release

Every repository entry must include a `rev:` field.

Bugs found in the existing `.pre-commit-config.yaml`:
- `charliermarsh/ruff-pre-commit` → wrong org, must be `astral-sh/ruff-pre-commit`
- `ruff-lint` → wrong hook id, must be `ruff`
- `check_yaml` → wrong hook id, must be `check-yaml` (hyphen not underscore)
- `black` repo was missing the `rev:` field entirely

---

## Solution

### Step 1: Open and edit the config file

```bash
nano /root/code/fraud-detection/.pre-commit-config.yaml
```

Fix the four bugs — the file should look like this:

```yaml
repos:
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v2.3.0
    hooks:
      - id: trailing-whitespace
      - id: end-of-file-fixer
      - id: check-yaml

  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.1.0
    hooks:
      - id: ruff

  - repo: https://github.com/psf/black-pre-commit-mirror
    rev: v0.1.0
    hooks:
      - id: black
```

Save with `Ctrl+O` → `Enter` → `Ctrl+X`

### Step 2: Auto-update all rev pins to latest releases

```bash
cd /root/code/fraud-detection
pre-commit autoupdate
```

This queries each referenced repository and rewrites the `rev:` pins to the latest released tag automatically.

### Step 3: Install the hooks and run against all files

```bash
pre-commit install
pre-commit run --all-files
```

## Key Takeaways

### 🪝 What is pre-commit?
`pre-commit` is a framework that runs automated checks **before every git commit**. When you run `git commit`, pre-commit intercepts it, runs all configured hooks against your staged files, and only allows the commit if everything passes. If a hook fails or modifies a file, the commit is blocked and you fix the issue first. This catches bugs, formatting issues, and bad practices before they ever enter the codebase.

### 🔄 Why pre-commit in MLOps?
In ML projects, code quality often takes a back seat to experimentation. Data scientists write notebooks and scripts quickly, and messy code ends up in production. Pre-commit enforces a quality gate automatically — no one has to remember to run `ruff` or `black` manually. Every commit is clean by construction, which makes code reviews faster and CI/CD pipelines more reliable.

### 📋 `.pre-commit-config.yaml` — how it works
The config file lists **repositories** (where hooks come from) and **hooks** (what to run). Each repo entry needs:
- `repo` — the GitHub URL of the hook provider
- `rev` — the exact version tag to use (pinned for reproducibility)
- `hooks` — list of hook IDs to enable from that repo

```yaml
- repo: https://github.com/astral-sh/ruff-pre-commit
  rev: v0.4.0        # pinned version
  hooks:
    - id: ruff       # hook to run
```

### 📌 Why pin with `rev:`?
Just like `requirements.txt` pins package versions, `rev:` pins the exact version of each hook. Without pinning, two teammates running `pre-commit` could get different versions of a hook that behave differently — breaking the "same result everywhere" guarantee. Always pin, and use `pre-commit autoupdate` to bump pins in a controlled way.

### ⚡ `pre-commit autoupdate`
Instead of manually looking up the latest version of each hook repo, `pre-commit autoupdate` queries GitHub and rewrites all `rev:` fields to the latest release tags automatically. This is the standard way to keep hooks up to date without hunting down version numbers by hand.

### ⚡ What ruff and black do as pre-commit hooks
In Day 006 we ran `ruff` and `black` manually from the terminal. Here they are wired into pre-commit, which means they run **automatically on every commit** without anyone having to remember.

- **`ruff`** runs as a linter hook — it checks your staged Python files for errors, unused imports, bad style, and unsorted imports. If it finds issues it cannot auto-fix, the commit is blocked until you fix them manually.
- **`black`** runs as a formatter hook — it checks that your code is formatted to black's standard. If any file would be reformatted, the commit is blocked and black rewrites the file for you. You just `git add` the reformatted file and commit again.

Together they form a two-layer quality gate on every single commit:
```
git commit
    → black  : is the code formatted?
    → ruff   : is the code clean and lint-free?
    → only then: commit is allowed
```

This is the standard pattern in production MLOps repos — linting and formatting are never optional, they are enforced at the git level.
