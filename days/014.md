# Day 014: Create a DVC Pipeline for Data Processing

**Category**: Data Version Control / ML Pipelines | **Difficulty**: 🟢 Beginner

---

## Problem

The xFusionCorp Industries ML team uses DVC pipelines to keep data processing reproducible. A draft `dvc.yaml` exists in the fraud-detection project, but `dvc repro` does not complete the full pipeline. Correct the pipeline definition so it runs cleanly end to end.

Given context:
1. A project exists at `/root/code/fraud-detection/` with DVC initialised. Python scripts are at `src/data/process_data.py` and `src/data/split_data.py`; raw input is at `data/raw/transactions.csv`. Do not modify the Python files or the input data.
2. The corrected pipeline must declare two stages:
   - `process_data` — depends on `data/raw/transactions.csv` and `src/data/process_data.py`; produces `data/processed/clean_transactions.csv`
   - `split_data` — depends on `data/processed/clean_transactions.csv` and `src/data/split_data.py`; produces `data/processed/train.csv` and `data/processed/test.csv`
3. After your changes, `dvc repro` must run end to end and `dvc status` must report no stale stages.

---

## Solution

### Step 1: Inspect the broken pipeline

```bash
cat /root/code/fraud-detection/dvc.yaml
```

### Step 2: Fix dvc.yaml with nano

```bash
nano /root/code/fraud-detection/dvc.yaml
```

Edit it to look exactly like this:

```yaml
stages:
  process_data:
    cmd: python src/data/process_data.py
    deps:
      - data/raw/transactions.csv
      - src/data/process_data.py
    outs:
      - data/processed/clean_transactions.csv
  split_data:
    cmd: python src/data/split_data.py
    deps:
      - data/processed/clean_transactions.csv
      - src/data/split_data.py
    outs:
      - data/processed/train.csv
      - data/processed/test.csv
```

Save with `Ctrl+O` → `Enter` → `Ctrl+X`

### Step 3: Run the pipeline

```bash
cd /root/code/fraud-detection
dvc repro
```

### Step 4: Verify no stale stages

```bash
dvc status
```

Should return: `Data and pipelines are up to date`

## Key Takeaways

### 🔁 What is a DVC pipeline?
A DVC pipeline is a sequence of stages defined in `dvc.yaml`, where each stage has a command to run, inputs it depends on (`deps`), and outputs it produces (`outs`). DVC tracks the relationship between stages and only reruns a stage when its inputs change — just like a smart build system, but for ML data pipelines.

### 📄 `dvc.yaml` structure
Every stage follows the same pattern:

```yaml
stages:
  stage_name:
    cmd: python script.py        # command to run
    deps:                        # inputs (trigger rerun if changed)
      - data/raw/input.csv
      - src/script.py
    outs:                        # outputs (tracked by DVC)
      - data/processed/output.csv
```

- `deps` — any file that, if changed, should trigger this stage to rerun
- `outs` — files produced by this stage, tracked by DVC like `dvc add`

### 🔗 Stage dependencies create a DAG
When `split_data` lists `data/processed/clean_transactions.csv` as a dep, and `process_data` lists it as an out, DVC automatically knows that `split_data` must run after `process_data`. This forms a **DAG (Directed Acyclic Graph)** — the execution order is determined by data dependencies, not by the order stages appear in the file.

```
data/raw/transactions.csv
        ↓
  process_data          (runs process_data.py)
        ↓
clean_transactions.csv
        ↓
   split_data           (runs split_data.py)
        ↓
train.csv + test.csv
```

### ⚡ `dvc repro` — smart re-execution
`dvc repro` is smarter than just running scripts sequentially. It checks the hash of every dep file and only reruns stages whose inputs have changed since the last run. If you run it twice in a row without changing anything, the second run does nothing — all stages are cached. This makes iterating on pipelines fast and reproducible.

### 📊 `dvc status` — pipeline health check
`dvc status` tells you which stages are stale (need to rerun) and which are up to date. In CI/CD, `dvc status` is often used as a check to verify the pipeline outputs match the committed state — if it reports stale stages, the pipeline hasn't been run against the current code.

### 🐛 File naming matters — exactly
The output filename in `dvc.yaml` must exactly match what the script actually produces. If the script writes `clean_transactions.csv` but `dvc.yaml` declares `clean_transaction.csv` (missing `s`), DVC runs the script successfully but then can't find the declared output and throws an error. Always verify filenames character by character.
