# Day 031: Train a Scikit-Learn Model with Reproducible Script

**Category**: Model Training | **Difficulty**: 🟡 Intermediate

---

## Problem

The xFusionCorp Industries ML platform team maintains a config-driven training pipeline so hyperparameters can be swapped without touching Python code. A training scaffold exists at `/root/code/fraud-detection/`, with the trainer already in place, but the YAML config has been left in a broken state and the pipeline won't run cleanly. The job is to fix the config, not the code.

1. The MLflow tracking server is already running on port `5000`, with an empty `fraud-detection` experiment in place.
2. The project layout under `/root/code/fraud-detection/`:
   - `data/train.csv` — a pre-generated 200-row synthetic binary classification dataset (columns: `amount`, `hour`, `num_tx_past_day`, `is_fraud`).
   - `src/models/train.py` — the config-driven trainer. Correct as-is, must not be modified.
   - `configs/train_config.yaml` — the project's training configuration. This file has bugs.
   - `models/` — where the serialised model must land.
3. Running `python3 /root/code/fraud-detection/src/models/train.py` currently either fails with an estimator or column error, or succeeds while dropping the model file outside the project tree. Find every setting in `train_config.yaml` that prevents a clean run and correct it.
4. The end state must include a successful run printed to stdout, exactly one new MLflow run in the `fraud-detection` experiment with the estimator's hyperparameters logged as run parameters, and a serialised model at `/root/code/fraud-detection/models/model.pkl`.

---

## Solution

### Step 1: Read the trainer to see what it actually expects

The script resolves the estimator class through a small registry — `RandomForestClassifier`, `GradientBoostingClassifier`, `LogisticRegression` — and fails fast with a clear message if the config names anything else. It also reads `target_column` straight from the config and will error out if that column doesn't exist in the CSV.

### Step 2: Read the broken config and spot the mismatches

Comparing it against what the trainer expects and what the dataset actually contains turns up three separate bugs:

1. `model.type: RandomForest` — not in the registry. The trainer wants the full sklearn class name, `RandomForestClassifier`.
2. `data.target_column: target` — the dataset's label column is actually `is_fraud`, not `target`.
3. `output.model_path: /root/code/model.pkl` — this writes the model outside the project tree entirely.

### Step 3: Fix the config in VS Code

```yaml
model:
  type: RandomForestClassifier
  n_estimators: 100
  max_depth: 5
  random_state: 42
data:
  train_path: /root/code/fraud-detection/data/train.csv
  target_column: is_fraud
output:
  model_path: /root/code/fraud-detection/models/model.pkl
mlflow:
  tracking_uri: http://localhost:5000
  experiment_name: fraud-detection
```

### Step 4: Run it

```bash
python3 /root/code/fraud-detection/src/models/train.py
```

```
accuracy=0.8000, f1_score=0.8261
model saved to /root/code/fraud-detection/models/model.pkl
🏃 View run upset-rook-216 at: http://localhost:5000/#/experiments/1/runs/430a50509d0745ffb1e3e2f513375773
```

One run, logged correctly, model landing in the right place.

## Key Takeaways

### Config-driven training means the bug almost never lives in the code
When a script takes everything from a YAML file and fails fast with descriptive errors (like this trainer does for an unknown estimator), the script itself is usually the wrong place to go looking. The error message — "unknown estimator type", "target column not found" — is already telling you exactly which config key is wrong. Read the script once to learn its contract, then debug the config against that contract.

### A registry pattern turns a typo into a clear error instead of a stack trace
Mapping config strings to actual classes through a dict (`ESTIMATOR_REGISTRY`) instead of doing something like `eval()` or `getattr()` means a bad value gets caught with a message that lists the valid options, right at the point of failure — not three layers down inside sklearn with a confusing `TypeError`.
