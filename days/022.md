# Day 022: Create and Organize MLflow Experiments

**Category**: ML Experimentation | **Difficulty**: 🟢 Beginner

---

## Problem

The xFusionCorp Industries ML platform team is onboarding two new ML projects and needs each one organised under its own MLflow experiment rather than sharing the `Default` experiment. Register both experiments through the MLflow tracking server and tag them with the owning team.

1. The MLflow tracking server is already running on port `5000`. One seeded experiment (`legacy-models`) is listed alongside the platform-created `Default` — both must not be modified.
2. Register two new experiments with the following metadata:
   - `fraud-detection` — tag key `team`, value `ml-platform`; non-empty description.
   - `churn-prediction` — tag key `team`, value `analytics`; non-empty description.

---

## Solution

### Step 1: Check existing experiments

```bash
python3 << 'EOF'
import mlflow
mlflow.set_tracking_uri("http://localhost:5000")
client = mlflow.tracking.MlflowClient()
for e in client.search_experiments():
    print(e.experiment_id, '|', e.name, '|', e.tags)
EOF
```

This revealed that both experiments had already been partially created in a prior attempt, so `create_experiment` raised `RESOURCE_ALREADY_EXISTS`. Always check first.

### Step 2: Add the team tags and descriptions

Since the experiments already existed, only the description tag was missing. Use `set_experiment_tag` with the reserved `mlflow.note.content` key to set the description:

```bash
python3 << 'EOF'
import mlflow
mlflow.set_tracking_uri("http://localhost:5000")
client = mlflow.tracking.MlflowClient()

client.set_experiment_tag("2", "mlflow.note.content", "Fraud detection project for xFusionCorp Industries")
client.set_experiment_tag("3", "mlflow.note.content", "Churn prediction project for xFusionCorp Industries")
print("Descriptions added.")
EOF
```

### Step 3: Verify the final state

```bash
python3 << 'EOF'
import mlflow
mlflow.set_tracking_uri("http://localhost:5000")
client = mlflow.tracking.MlflowClient()
for e in client.search_experiments():
    print(e.experiment_id, '|', e.name, '|', e.tags)
EOF
```

Expected output:

```
3 | churn-prediction | {'team': 'analytics', 'mlflow.note.content': 'Churn prediction project...'}
2 | fraud-detection  | {'team': 'ml-platform', 'mlflow.note.content': 'Fraud detection project...'}
1 | legacy-models    | {'team': 'ml-legacy', 'status': 'archived', 'mlflow.note.content': '...'}
0 | Default          | {}
```

## Key Takeaways

### 🧪 Experiments as project namespaces
MLflow experiments are the top-level organisational unit. Every run belongs to exactly one experiment. Keeping each project in its own experiment makes it easy to filter runs, compare metrics, and manage access — without runs from different projects polluting the same list.

### 🏷️ Experiment-level tags
Tags on experiments (not runs) are key-value metadata about the project itself — who owns it, its status, its domain. Setting `team=ml-platform` on `fraud-detection` makes ownership queryable via `client.search_experiments(filter_string="tag.team = 'ml-platform'")` without opening the UI.

### 📝 `mlflow.note.content` — the description tag
MLflow does not have a dedicated description field on experiments. The convention is to use the reserved tag key `mlflow.note.content` — the UI renders its value as the experiment description. Set it with `client.set_experiment_tag(experiment_id, "mlflow.note.content", "...")`.

### ⚠️ `RESOURCE_ALREADY_EXISTS` — check before creating
`mlflow.create_experiment()` raises a `RestException` if the name is already taken. Always call `client.search_experiments()` first to check the current state, then decide whether to create or just update tags on the existing record.

### 🔌 `MlflowClient` vs the fluent API
The fluent API (`mlflow.create_experiment`, `mlflow.set_experiment`) is convenient for use inside training scripts. The `MlflowClient` gives finer-grained control — setting tags on specific experiment IDs, searching with filters, and managing runs programmatically — making it the right tool for admin and setup tasks like this one.
