# Day 019: Build a Complete DVC ML Pipeline with Remote Storage

**Category**: ML Experimentation | **Difficulty**: 🟡 Intermediate

---

## Problem

The xFusionCorp Industries ML team needs to complete a production-ready DVC pipeline for a fraud-detection project. Three stages are already wired in `dvc.yaml`, two remain, and the pipeline must finish as a reproducible, SeaweedFS-backed, `v1.0`-tagged release.

1. A project exists at `/root/code/ml-pipeline/` with Git and DVC initialised. The `params.yaml` is in place and `.dvc/config` is pre-configured to push to the SeaweedFS bucket `dvc-storage` at `http://localhost:8333`.
2. The `ingest`, `validate`, and `preprocess` stages are already declared in `dvc.yaml`, but one of them contains an incorrect output path that prevents `dvc repro` from completing. Find and fix it.
3. The remaining two stages need to be added:
   - `train` — Depends on the preprocessed dataset and `scripts/train.py`; reads `n_estimators`, `max_depth`, `test_size`, and `random_seed` from `params.yaml`; outputs `models/model.pkl` and `data/processed/test_split.csv`; declares `metrics.json` as a DVC metric with `cache: false`.
   - `evaluate` — Depends on `models/model.pkl`, `data/processed/test_split.csv`, and `scripts/evaluate.py`; outputs `reports/evaluation.json` declared with `cache: false`.
4. The two scripts needed are pre-staged at `/root/code/ml-pipeline/scripts-staging/train.py` and `scripts-staging/evaluate.py`. Copy them into `scripts/` before adding the stages.
5. Run the full pipeline with `dvc repro`, push the cache to the SeaweedFS remote with `dvc push`, and tag the current state as `v1.0`.
6. Commit every change to Git so the release is fully captured.

---

## Solution

### Step 1: Inspect the existing pipeline and scripts

```bash
cd /root/code/ml-pipeline
cat dvc.yaml
cat params.yaml
cat .dvc/config
```

Then identify the broken output path:

```bash
cat scripts/preprocess.py | grep -i "to_csv\|output"
```

The script writes to `data/processed/clean.csv` but `dvc.yaml` declared `data/processed/cleaned.csv` — one character difference that breaks the dependency chain.

### Step 2: Confirm the staging script paths

```bash
cat scripts-staging/train.py | grep -i "to_csv\|pkl\|json\|read_csv\|open"
cat scripts-staging/evaluate.py | grep -i "to_csv\|pkl\|json\|read_csv\|open"
```

This reveals:
- `train.py` reads `data/processed/clean.csv`, outputs `models/model.pkl`, `data/processed/test_split.csv`, and `metrics.json`
- `evaluate.py` reads `models/model.pkl` and `data/processed/test_split.csv`, outputs `reports/evaluation.json`

### Step 3: Copy the staging scripts

```bash
cp scripts-staging/train.py scripts/train.py
cp scripts-staging/evaluate.py scripts/evaluate.py
```

### Step 4: Rewrite `dvc.yaml` with the fix and two new stages

```bash
cat > dvc.yaml << 'EOF'
stages:
  ingest:
    cmd: python scripts/ingest.py
    deps:
      - scripts/ingest.py
      - data/raw/data.csv
  validate:
    cmd: python scripts/validate.py
    deps:
      - data/raw/data.csv
      - scripts/validate.py
    outs:
      - reports/validation.json:
          cache: false
  preprocess:
    cmd: python scripts/preprocess.py
    deps:
      - data/raw/data.csv
      - scripts/preprocess.py
    outs:
      - data/processed/clean.csv
  train:
    cmd: python scripts/train.py
    deps:
      - data/processed/clean.csv
      - scripts/train.py
    params:
      - n_estimators
      - max_depth
      - test_size
      - random_seed
    outs:
      - models/model.pkl
      - data/processed/test_split.csv
    metrics:
      - metrics.json:
          cache: false
  evaluate:
    cmd: python scripts/evaluate.py
    deps:
      - models/model.pkl
      - data/processed/test_split.csv
      - scripts/evaluate.py
    outs:
      - reports/evaluation.json:
          cache: false
EOF
```

### Step 5: Run the full pipeline

```bash
dvc repro
```

All five stages run in order:

```
Running stage 'ingest':    Data ingested successfully: 20 rows, 5 columns
Running stage 'validate':  Validation: 20 rows, valid=True
Running stage 'preprocess': Preprocessed: 20 clean rows
Running stage 'train':     Trained: {'accuracy': 1.0, 'f1_score': 1.0}
Running stage 'evaluate':  Evaluation: {'accuracy': 1.0, 'f1_score': 1.0, ...}
```

### Step 6: Push the cache to the SeaweedFS remote

```bash
dvc push
```

Output confirms 3 files pushed to the `s3://dvc-storage` bucket.

### Step 7: Commit all changes and tag as `v1.0`

```bash
git add dvc.yaml dvc.lock scripts/train.py scripts/evaluate.py
git commit -m "Complete pipeline: fix preprocess path, add train and evaluate stages"
git tag v1.0
```

## Key Takeaways

### 🔍 Debugging broken DVC pipelines
When `dvc repro` fails silently or skips stages unexpectedly, the first thing to check is whether the `outs` path in `dvc.yaml` exactly matches what the script actually writes to disk. A single character difference (`cleaned.csv` vs `clean.csv`) breaks the dependency chain for every downstream stage.

### 🏗️ Anatomy of a `dvc.yaml` stage
Each stage has four optional but important keys:
- `deps` — files or directories this stage reads; any change triggers a rerun
- `params` — keys from `params.yaml` this stage reads; tracked separately so parameter sweeps are lightweight
- `outs` — files this stage writes; DVC caches these by hash
- `metrics` — like `outs` but with `cache: false` so the file stays in Git and is readable by `dvc metrics show`

### 📦 SeaweedFS as a DVC remote
DVC supports any S3-compatible object store as a remote. SeaweedFS exposes an S3 API, so configuring it only requires setting `endpointurl` alongside the standard `url`, `access_key_id`, and `secret_access_key` in `.dvc/config`. `dvc push` then uploads all cached artefacts to the bucket under a content-addressed `files/md5/...` layout.

### 🏷️ Tagging a pipeline release with Git
Combining `git tag v1.0` with a committed `dvc.lock` creates a fully reproducible snapshot. Anyone with access to the repo and the DVC remote can run `git checkout v1.0 && dvc pull` to get the exact data, model, and metrics that correspond to that release — no manual file management needed.
