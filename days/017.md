# Day 017: Run and Compare DVC Experiments

**Category**: ML Experimentation | **Difficulty**: 🟢 Beginner

---

## Problem

The xFusionCorp Industries data science team compares multiple training runs with different hyperparameters using DVC experiments. Run three experiments that vary the `n_estimators` hyperparameter, identify the best-performing one, and promote it to the tracked workspace.

1. `params.yaml` contains `n_estimators: 100` and the baseline pipeline has been run once.
2. Run three DVC experiments with `n_estimators` set to `50`, `200`, and `500`.
3. Compare the experiments and choose the one whose `f1_score` is the highest.
4. Apply the chosen experiment to the workspace so its `n_estimators`, `metrics.json`, and `models/model.pkl` become the tracked state.

---

## Solution

### Step 1: Run the three experiments

```bash
cd /root/code/fraud-detection
dvc exp run --set-param n_estimators=50
dvc exp run --set-param n_estimators=200
dvc exp run --set-param n_estimators=500
```

### Step 2: Compare all experiments

```bash
dvc exp show
```

### Step 3: Apply the best experiment

```bash
dvc exp apply pesky-tads
```

## Key Takeaways

### 🧪 What is a DVC experiment?
A DVC experiment is a lightweight run of your pipeline with modified parameters, without creating a new Git commit for each run. DVC tracks the parameters, metrics, and outputs of each run in the background so you can compare them later and only commit the winner to Git.

### ⚡ `dvc exp run --set-param`
This command runs the full pipeline with a parameter overridden on the fly — no need to edit `params.yaml` manually each time:
```bash
dvc exp run --set-param n_estimators=200
```
Only the stages affected by the parameter change rerun. In our case, `process_data` and `split_data` were skipped every time — only `train` reran.

### 📊 `dvc exp show` — the comparison table
`dvc exp show` displays all experiments in a table with their parameters and metrics side by side. This makes it easy to spot which configuration produced the best results without digging through log files.

### 🏆 `dvc exp apply` — promoting the winner
`dvc exp apply <name>` applies the chosen experiment's outputs back to the workspace — updating `params.yaml`, `metrics.json`, `dvc.lock`, and `models/model.pkl` to match that run. After applying, you commit to Git to make it the new tracked state.
