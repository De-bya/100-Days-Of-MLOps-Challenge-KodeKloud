# Day 020: Install and Start the MLflow Tracking Server

**Category**: ML Experimentation | **Difficulty**: 🟢 Beginner

---

## Problem

The xFusionCorp Industries ML team is adopting MLflow for experiment tracking. Bring up a local MLflow tracking server on the ML pipeline workstation so experiments can be logged from the team's training code.

MLflow 3.x is pre-installed on the controlplane. Launch the tracking server in the background so that every end-state requirement below holds.

1. The server is listening on port `5000` and is reachable on all interfaces.
2. The backend store is a SQLite database at `/root/code/mlflow-backend/mlflow.db`. The database file must exist after the server has started.
3. The artifact root is `/root/code/mlflow-artifacts/`.
4. Any parent directories the server needs must be in place before it starts — MLflow will abort if the backend directory is missing.
5. The MLflow UI button at the top of the lab must open a responsive dashboard in the browser. The button routes through the lab proxy, so the server must accept requests from any origin (`--cors-allowed-origins '*'`) and any host header (`--allowed-hosts '*'`) to avoid proxy-related rejections.
6. The server process must persist in the background so it survives terminal closure.

---

## Solution

### Step 1: Create the required directories

MLflow will not create the backend directory automatically — if it is missing the server aborts on startup.

```bash
mkdir -p /root/code/mlflow-backend
mkdir -p /root/code/mlflow-artifacts
```

### Step 2: Launch the MLflow server in the background

```bash
nohup mlflow server \
  --backend-store-uri sqlite:////root/code/mlflow-backend/mlflow.db \
  --artifacts-destination /root/code/mlflow-artifacts \
  --host 0.0.0.0 \
  --port 5000 \
  --cors-allowed-origins '*' \
  --allowed-hosts '*' \
  > /root/code/mlflow-backend/mlflow.log 2>&1 &
```

The shell prints the background PID (e.g. `[1] 2775`) confirming the process started.

### Step 3: Verify the server is healthy

```bash
sleep 3
curl -s http://localhost:5000/health
```

Expected output:

```
OK
```

The SQLite database file is also created automatically on first startup:

```bash
ls /root/code/mlflow-backend/mlflow.db
```

---

## Key Takeaways

### 🗄️ `--backend-store-uri` — where metadata lives
MLflow stores all experiment metadata (run IDs, parameters, metrics, tags) in a backend store. Using `sqlite:////path/to/mlflow.db` tells MLflow to use a local SQLite file. The four slashes are required: three for the URI scheme (`sqlite:///`) plus one for the absolute path (`/root/...`). For production deployments this is typically swapped for a PostgreSQL or MySQL URI.

### 📦 `--artifacts-destination` — where model files live
Artifacts (model binaries, plots, datasets) are stored separately from metadata. Setting `--artifacts-destination` to a local directory works fine for single-machine setups. In a team environment this would point to an S3 bucket, GCS, or Azure Blob Storage so all machines share the same artifact store.

### 🌐 `--host 0.0.0.0` — listen on all interfaces
By default MLflow binds to `127.0.0.1` (localhost only). Passing `0.0.0.0` makes it reachable from any network interface — required here so the lab proxy can forward browser traffic to the server.
