# Day 009: Create a Custom ML Project Template with Cookiecutter

**Category**: ML Project Templating | **Difficulty**: 🟢 Beginner 

---

## Problem

The xFusionCorp Industries platform team maintains a Cookiecutter template that new ML projects are generated from. A draft template exists at `/root/code/mlops-template/`, but it does not render correctly. Correct the template and use it to generate a project.

The corrected template must satisfy:
- `cookiecutter.json` declares four variables: `project_name` (default `my-ml-project`), `author` (default `xFusionCorp`), `python_version` (default `3.11`), and `ml_framework` with choices `sklearn`, `pytorch`, `tensorflow`
- `requirements.txt` logic: contains `scikit-learn` when `ml_framework` is `sklearn`, `torch` when `pytorch`, `tensorflow` when `tensorflow`
- `README.md` must reference both `project_name` and `author` from cookiecutter variables
- The template directory must contain files `README.md` and `requirements.txt` and directories `data/`, `models/`, `src/`, and `tests/`

---

## Solution

### Step 1: Inspect the existing template

```bash
find /root/code/mlops-template -type f | sort
cat /root/code/mlops-template/cookiecutter.json
cat "/root/code/mlops-template/{{cookiecutter.project_name}}/README.md"
cat "/root/code/mlops-template/{{cookiecutter.project_name}}/requirements.txt"
```

### Step 2: Fix cookiecutter.json

```bash
nano /root/code/mlops-template/cookiecutter.json
```

Edit it to:

```json
{
    "project_name": "my-ml-project",
    "author": "xFusionCorp",
    "python_version": "3.11",
    "ml_framework": ["sklearn", "pytorch", "tensorflow"]
}
```

Save with `Ctrl+O` → `Enter` → `Ctrl+X`

### Step 3: Fix README.md

```bash
nano "/root/code/mlops-template/{{cookiecutter.project_name}}/README.md"
```

Edit it to:

```
# {{cookiecutter.project_name}}
Created by {{cookiecutter.author}}.
```

Save with `Ctrl+O` → `Enter` → `Ctrl+X`

### Step 4: Fix requirements.txt

```bash
nano "/root/code/mlops-template/{{cookiecutter.project_name}}/requirements.txt"
```

Edit it to:

```
{% if cookiecutter.ml_framework == 'sklearn' %}
scikit-learn
{% elif cookiecutter.ml_framework == 'pytorch' %}
torch
{% elif cookiecutter.ml_framework == 'tensorflow' %}
tensorflow
{% endif %}
```

Save with `Ctrl+O` → `Enter` → `Ctrl+X`

### Step 5: Generate the project

```bash
cookiecutter /root/code/mlops-template/ -o /root/code/ --no-input project_name=churn-model ml_framework=sklearn
```

### Step 6: Verify

```bash
cat /root/code/churn-model/requirements.txt
cat /root/code/churn-model/README.md
```

## Key Takeaways

### 🍪 What is Cookiecutter?
Cookiecutter is a command-line tool that generates projects from templates. You define a template once — folder structure, file contents, variable placeholders — and anyone on the team can spin up a new standardized project in seconds with a single command. Instead of copying an old project and manually renaming things, you run `cookiecutter <template>` and get a clean, consistent project every time.

### 🏗️ Why project templates in MLOps?
Every ML project at a company tends to need the same structure: `data/`, `src/`, `models/`, `tests/`, a `requirements.txt`, a `README.md`. Without a template, each data scientist creates their own layout, and the team ends up with inconsistent projects that are hard to navigate, automate, or deploy. A Cookiecutter template enforces the standard layout across every new project from day one.

### 📋 `cookiecutter.json` — the variable definitions
This file defines all the variables the template uses, along with their default values. When you run cookiecutter interactively, it prompts you for each variable. With `--no-input`, it uses the defaults or values you pass on the command line.

```json
{
    "project_name": "my-ml-project",
    "author": "xFusionCorp",
    "ml_framework": ["sklearn", "pytorch", "tensorflow"]
}
```

When `ml_framework` is a list, the first item is the default and cookiecutter presents the rest as choices.

### 🧩 Jinja2 templating — `{{` and `{%`
Cookiecutter uses Jinja2 as its templating engine — the same engine used by tools like Ansible, Flask, and Airflow. Two syntax patterns matter:

| Syntax | Purpose | Example |
|--------|---------|---------|
| `{{ variable }}` | Insert a value | `{{ cookiecutter.project_name }}` |
| `{% if/elif/endif %}` | Conditional logic | `{% if cookiecutter.ml_framework == 'sklearn' %}` |

Note: Jinja2 uses `==` for comparison, not `=`. Using a single `=` throws a `TemplateSyntaxError`.

### 🔡 Variable names are case-sensitive
`{{ cookiecutter.Author }}` and `{{ cookiecutter.author }}` are completely different variables. If `cookiecutter.json` defines `author` (lowercase) but the template uses `Author` (uppercase), it renders as an empty string or throws an error. Always match the exact case defined in `cookiecutter.json`.

### ⚙️ `--no-input` and passing variables on the command line
The `--no-input` flag skips the interactive prompts and uses defaults from `cookiecutter.json`. You can override specific variables by passing them as `key=value` arguments:
```bash
cookiecutter /root/code/mlops-template/ -o /root/code/ --no-input project_name=churn-model ml_framework=sklearn
```
This is essential for automated pipelines where you can't have interactive prompts.
